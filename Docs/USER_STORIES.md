# User Stories
## User Stories
User stories describe the key actions users must be able to perform in Uni‑Bridge.

US.1 – Register/Login  
As a user, I want to create an account and log in so that I can access the platform.

US.2 – Add/Edit Modules  
As a user, I want to add or edit the modules I can teach or want to learn so that the system can match me with the right people.

US.3 – Search for Experts  
As a learner, I want to search for experts by module so that I can find someone who can help me.

US.4 – Send/Accept/Decline Requests  
As a user, I want to send, accept, or decline meeting requests so that I can manage who I connect with.

US.5 – View Profiles  
As a learner, I want to view a teacher’s profile so that I can decide whether to send a request.

US.6 – View Request Status  
As a user, I want to see the status of my sent and received requests so that I know what actions to take next.

US.7 – Receive Notifications  
As a user, I want to receive notifications so that I am aware of new requests and updates.

## Prioritised User Stories Table
|ID	|User Story	|Priority|	Reason|
|---|------|------|----|
|US.1|	Register/Login	|Must Have	|Users cannot access the platform without authentication.|
|US.2	|Add/Edit Modules	|Must Have	|Required for matching learners with experts.|
|US.3	|Search for Experts|	Must Have	|Core purpose of the platform.|
|US.4	|Send/Accept/Decline Requests|	Must Have	|Enables the main interaction between users.|
|US.5	|View Profiles	|Should Have	|Helps users make informed decisions.|
|US.6	|View Request Status	|Should Have	|Users need visibility of their interactions.|
|US.7	|Receive Notifications	|Could Have	|Useful enhancement but not essential for MVP.|

## MoSCoW Prioritisation Rationale
The team used the MoSCoW method to prioritise features for the MVP:

Must Have: Authentication, module selection, search, and request handling.

Should Have: Profile viewing and request status tracking.

Could Have: Notifications and other enhancements that can be added in later sprints.
